# 强势股模式研究与在线匹配设计

## 1. 目标

自动研究最近一周、一月、一季度和一年强势股在启动前的共同模式，并使用经过样本外验证的模式对当日股票进行匹配和分层。

系统需要回答：

- 历史强势股在上涨前具备哪些价格、量能、波动、相对强度和板块特征？
- 不同上涨类型是否由不同模式驱动？
- 当前股票与哪些历史模式相似？
- 这些模式在未知时间段、不同市场状态和交易成本后是否仍然有效？
- 当前匹配为什么成立，什么情况下失效？

## 2. 设计原则

- 不把所有上涨股票混成一个模型。
- 不只研究当前赢家，必须建立历史滚动样本。
- 不使用随机训练测试拆分，必须按时间 Walk-forward。
- 不只看命中样本，必须包括普通股票和失败样本。
- 不允许训练代码直接决定线上交易。
- 所有模式必须能还原为特征范围、支持样本和统计结果。
- 模式发布后保持不可变；调整产生新版本。

## 3. 周期定义

第一版固定四个观察周期：

| 周期 ID | 观察窗口 | 启动前特征窗口 | 预期持有参考 |
|---|---:|---:|---:|
| `week` | 5 个交易日 | 5、10、20 日 | 2-10 日 |
| `month` | 20 个交易日 | 10、20、60 日 | 5-30 日 |
| `quarter` | 60 个交易日 | 20、60、120 日 | 20-90 日 |
| `year` | 250 个交易日 | 60、120、250 日 | 60-250 日 |

观察窗口和特征窗口按交易日计算，不使用自然日近似。

## 4. 强势标签

### 4.1 初始候选

每个历史基准日 `t`，计算未来观察窗口内：

- 区间收益。
- 相对基准指数收益。
- 相对所属行业收益。
- 最大有利波动。
- 最大回撤。
- 上涨连续性。
- 成交额可交易性。

### 4.2 强势评分

强势股不是纯涨幅榜。建议使用可配置的横截面评分：

```text
strength_score =
    return_percentile
  + benchmark_excess_percentile
  + industry_excess_percentile
  + trend_consistency_percentile
  - drawdown_percentile
  - illiquidity_penalty
  - untradeable_limit_penalty
```

第一版可将各项归一化到 0-100，再由配置指定权重。

### 4.3 正样本

同时满足：

- `strength_score` 位于当期可交易股票前 5%-10%。
- 未来区间收益为正。
- 未来相对基准收益达到最低门槛。
- 不属于无法真实买入的一字连续涨停样本。
- 基准日前已有足够历史数据。

### 4.4 对照组

至少包括：

1. 同行业、同市值和相近流动性的普通股票。
2. 启动形态相似但未来未上涨的失败样本。
3. 未来显著跑输指数的负样本。
4. 突破后快速跌回结构下方的假突破样本。

对照组用于判断某特征是否真的区分赢家，而不是所有股票都常见。

### 4.5 防止未来泄漏

基准日 `t` 的输入只能使用 `t` 收盘及以前可获得的数据。

禁止：

- 用未来行业分类修正历史样本。
- 用未来退市结果提前剔除股票。
- 用未来成交额判断当时可交易性。
- 用全样本均值标准化过去特征。
- 在聚类前使用测试期标签调参。

## 5. 模式类型

聚类前先通过特征族和后验解释将强势股划分为五类：

### 5.1 连续趋势型

典型特征：

- 中短期相对强度持续为正。
- 均线斜率稳定。
- 回撤浅。
- 上涨日量能高于回调日。
- 收盘位置稳定偏强。

### 5.2 波动收缩突破型

典型特征：

- 短期波动低于中期波动。
- 成交量收缩。
- 平台宽度受控。
- 突破日成交额和收盘质量改善。

### 5.3 板块龙头型

典型特征：

- 所属行业或概念处于市场前列。
- 个股在板块内相对收益、成交额和创新高能力领先。
- 板块宽度支持，而非单票孤立上涨。

### 5.4 超跌反转型

典型特征：

- 中期回撤明显。
- 下跌波动开始收敛。
- 卖压量能衰减。
- 价格重新站上短期结构。
- 相对弱势停止恶化。

### 5.5 事件驱动型

典型特征：

- 启动附近存在高可信事件。
- 事件涉及行业或公司实体。
- 价格、成交额和板块强度在事件后确认。
- 事件并非已充分提前交易。

事件类型只作为独立特征族，不覆盖价格和风险条件。

## 6. 特征体系

### 6.1 价格与趋势

- 1、5、10、20、60、120、250 日收益。
- 相对 MA5/10/20/60/120/250 偏离。
- 均线排序和斜率。
- 线性回归斜率与拟合优度。
- 距窗口高低点的位置。
- 创新高频率。
- 收盘位置和上下影比例。

### 6.2 波动与回撤

- ATR 百分比。
- 已实现波动率。
- 5/20、20/60 波动收缩比。
- 窗口最大回撤。
- 回撤持续时间。
- 跳空频率。
- 极端 K 线比例。

### 6.3 成交与流动性

- 量比和成交额比。
- 5/20、20/60 平均成交额变化。
- 上涨日与下跌日成交量比。
- 换手率及变化。
- 成交额横截面分位。
- 量价相关性。

### 6.4 相对强度

- 相对主要指数收益。
- 相对行业收益。
- 行业内收益分位。
- 行业内成交额分位。
- 行业强度和行业宽度。

### 6.5 结构与形态

- Donchian 突破距离。
- 平台宽度。
- 支撑和阻力触碰次数。
- 假跌破收回。
- 回踩 MA10/MA20 的 ATR 距离。
- 现有原始信号和短中长票池命中，作为辅助证据。

### 6.6 市场环境

- 指数高于 MA20/60 的状态。
- 全市场 MA20 以上股票比例。
- 新高新低比。
- 涨停、跌停和炸板统计。
- 市场成交额变化。
- 市场波动状态。

### 6.7 事件特征

- 相关事件数量。
- 最高来源可信度。
- 新颖度。
- 事件方向。
- 预期时间跨度。
- 行情确认分。
- 事件拥挤或已交易程度。

## 7. 特征快照

Rust 每个交易日收盘后生成不可变特征快照。

建议表：

### `analysis_feature_snapshots`

```text
trade_date
code
feature_version
features JSONB
input_fingerprint
created_at
PRIMARY KEY (trade_date, code, feature_version)
```

第一版可以使用 JSONB 保持迭代速度，但高频筛选字段应同步保存为类型化列或物化视图。禁止在线匹配时重新拼接五年历史 SQL。

特征快照必须保存：

- 缺失特征列表。
- 数据最新日期。
- 输入行数。
- 计算逻辑版本。

## 8. 研究 Worker

### 8.1 运行方式

采用 Python 批处理 CLI：

```text
python -m qbot_research.cli train --as-of 2026-07-10 --horizon week
python -m qbot_research.cli train-all --as-of 2026-07-10
python -m qbot_research.cli validate --run-id ...
```

Rust 调度器可以启动进程，也可以由系统定时任务启动。第一版不做 Python 常驻服务。

### 8.2 聚类流程

1. 从特征快照读取历史样本。
2. 仅使用训练窗口拟合缺失值和缩放器。
3. 按主要模式类型或特征族分层。
4. 对正样本进行候选聚类。
5. 对聚类进行稳定性和可解释性筛选。
6. 使用对照组计算区分能力。
7. 在未来验证窗口计算交易结果。
8. 导出候选模式。

候选算法可比较：

- K-Means 或 MiniBatch K-Means：简单、稳定、容易上线。
- Gaussian Mixture：允许软匹配。
- HDBSCAN：发现非球形簇并识别噪音，但版本稳定性要求更高。

第一版推荐：标准化特征 + K-Means/GMM 双模型比较。只有两个算法都支持、且跨窗口稳定的模式优先进入审核。

### 8.3 模式解释

每个聚类必须转换为：

- 高贡献特征。
- 特征中位数和分位区间。
- 与对照组的差异。
- 典型历史案例。
- 反例和失败案例。
- 可读的必要条件和风险条件。

禁止只保存不可解释的聚类中心。

## 9. Walk-forward 验证

训练范围为滚动最近五年。

示例：

```text
训练 24 个月 -> 验证 6 个月
窗口向前滚动 3 个月
重复至数据截止日
```

四个周期可以使用不同训练长度，但必须保证：

- 测试窗口晚于训练窗口。
- 同一股票相邻样本避免泄漏到训练和测试两侧。
- 标准化器只用训练数据拟合。
- 参数选择不得查看最终保留测试期。

### 9.1 验证指标

- 样本数。
- 相对基准收益。
- 成本后平均和中位数收益。
- 胜率。
- 盈亏比。
- Profit Factor。
- 最大回撤。
- 最大连续亏损。
- 持有期分布。
- 换手率。
- 不同市场状态表现。
- 不同年份表现。
- 模式漂移。

### 9.2 发布门槛

第一版建议至少满足：

- 独立验证样本数达到配置门槛。
- 多数 Walk-forward 窗口为正超额收益。
- 成本后期望收益为正。
- 最大回撤低于策略限制。
- 特征方向在不同窗口不发生根本反转。
- 模式不是由少数极端样本贡献全部收益。

具体数值放入配置和版本记录，不硬编码在业务逻辑中。

## 10. 模式存储

### 10.1 `analysis_training_runs`

记录：

- 运行 ID。
- 截止日期。
- 周期。
- 数据范围。
- 配置。
- 状态。
- 日志摘要。
- 输入指纹。
- 代码版本。

### 10.2 `analysis_pattern_versions`

记录：

```text
pattern_version_id
pattern_id
horizon
pattern_type
status
schema_version
feature_version
logic_version
model_payload JSONB
validation_payload JSONB
trained_from
trained_until
approved_by
published_at
created_at
```

### 10.3 `analysis_pattern_examples`

保存每个模式的典型正例、失败例和相似度，供审核和解释。

### 10.4 `analysis_pattern_sets`

一个线上发布集合包含四个周期的若干模式版本。在线任务按集合 ID 加载，保证一次运行中不会混用不同发布时间的模式。

## 11. Rust 在线匹配

### 11.1 输入

- 当日特征快照。
- 指定的已发布模式集合。
- 市场环境。
- 当日有效事件因子。
- 证券可交易规则。

### 11.2 相似度

支持：

- 标准化欧氏或 Mahalanobis 距离。
- GMM 概率。
- 必要条件硬过滤。
- 风险条件扣分。

线上计算公式必须由模型 payload 完整描述。Rust 不得隐式复制 Python 内部默认值。

### 11.3 最终评分

```text
final_score =
    pattern_similarity
  + relative_strength_score
  + sector_confirmation_score
  + market_regime_score
  + event_confirmation_score
  - extension_penalty
  - liquidity_penalty
  - concentration_penalty
  - data_quality_penalty
```

事件确认分第一版默认不超过总分的 10%，并且只有存在价格和板块确认时才能为正。

### 11.4 输出

`PatternCandidate`：

```text
trade_date
code
name
horizon
pattern_version_id
pattern_type
similarity_score
final_score
tier
matched_features
risk_flags
supporting_signals
event_refs
invalidations
input_fingerprint
```

分层：

- A：模式和环境均确认，风险受控。
- B：模式匹配，但存在一个需要观察的风险。
- Watch：局部匹配或事件尚未获行情确认。
- Reject：硬过滤失败。

## 12. 与现有扫描系统的关系

现有 `src/signals/` 保留，用作：

- 可解释辅助特征。
- 历史统计对照。
- 候选说明。

新的模式候选不应伪装成第 25、26 个普通布尔信号。建议新增独立输出族：

```text
pattern_week_a
pattern_week_b
pattern_month_a
pattern_month_b
pattern_quarter_a
pattern_quarter_b
pattern_year_a
pattern_year_b
```

长期可以由新的模式层替代部分重复原始信号，但第一版保持兼容。

## 13. 错误处理

- 当日特征不完整：候选降级或拒绝，并记录缺失项。
- 模式版本不可解析：整个模式集合拒绝加载。
- 无已发布模式：跳过模式选股，不影响旧扫描。
- 研究任务超时：保留旧发布版本。
- 样本不足：模式保持 draft，不允许发布。
- 市场状态未知：不加市场奖励，并增加未知风险标记。

## 14. 测试

### 14.1 标签测试

- 未来窗口边界正确。
- 退市、停牌和上市天数处理正确。
- 一字板不可交易样本不被错误计入。
- 对照组不泄漏未来结果。

### 14.2 特征测试

- 每个特征只读基准日以前数据。
- Rust 和 Python 对固定 fixture 计算一致。
- 缺失值行为固定。
- 市场和行业数据按日期正确对齐。

### 14.3 模型测试

- 同一随机种子和数据得到一致结果。
- 不同窗口模式 ID 不碰撞。
- 模式导出 Schema 完整。
- Rust 拒绝未知 Schema。

### 14.4 匹配测试

- 典型正例进入正确模式。
- 失败样本被风险条件扣分。
- 事件不能单独把 Reject 提升为 A。
- 市场数据不完整时安全降级。

## 15. 第一版验收标准

- 能构造最近五年的四周期滚动样本。
- 能生成强势股和三类以上对照样本。
- 能训练并保存候选模式版本。
- 能输出 Walk-forward 验证报告。
- 模式必须经过人工发布门槛。
- Rust 能加载发布模式并对当日全市场匹配。
- 每个候选有可读匹配特征和失效条件。
- 研究任务失败不会影响现有扫描和日报。
