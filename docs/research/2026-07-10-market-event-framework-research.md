# 市场事件采集与分析框架调研

> 调研日期：2026-07-10
> 目的：在实现 QBot 市场事件分析模块前，比较成熟商业系统、开放数据平台和研究型框架，提取可复用的架构原则。

## 1. 结论摘要

成熟市场事件系统没有依赖单一模型完成全部工作。常见生产链路是：

```text
多来源采集
-> 原始证据持久化
-> 精确与近重复检测
-> 实体识别和消歧
-> 文章级分类、情绪和时间标注
-> 跨文章事件聚类
-> 事件级相关性、新颖度和重要性评分
-> 摘要和代表性证据选择
-> 市场影响验证
-> 告警、搜索、研究或交易因子
```

调研得到的核心判断：

1. **GDELT** 最适合开放的全球宏观、地缘和舆情补充，不适合作为 A 股公司级事实的唯一来源。
2. **Event Registry / NewsAPI.ai** 最值得借鉴的是“文章与事件分层”、近重复检测、实体消歧和跨来源事件聚类。
3. **RavenPack** 最值得借鉴的是对每个“事件—实体”关系分别计算相关性、新颖度、时间和影响，而不是只给整篇新闻一个情绪分。
4. **Bloomberg NSTM** 展示了成熟的双层聚类架构：采集时做低成本在线聚类，查询或批处理时再精细合并；事件重要性还应考虑来源多样性，而不只是报道数量。
5. **Dataminr 类系统** 说明事件是持续演化的对象，需要 `first_seen`、`last_seen`、状态、严重度和更新记录，而不是一次性静态新闻摘要。
6. **Doc2EDAG 等中文金融事件抽取框架** 说明公告中的事件参数往往跨句分散，正式公告必须采用文档级抽取，不能只做标题或单句分类。
7. **因果知识图谱研究** 支持“模型抽取 + 确定性本体和规则”的混合方案。纯 LLM 自动生成知识图谱容易缺少一致性与可解释性。
8. 事件是否影响市场，应通过基准调整后的异常收益、行业相对表现、板块宽度和成交验证，而不是简单查看相关股票当天是否上涨。
9. QBot 当前规模不需要先引入 Kafka、Flink 或图数据库。PostgreSQL、可重入批任务和清晰模块接口足够完成第一版。

## 2. 评估维度

本次按以下能力评估：

- 来源覆盖和数据许可。
- 延迟和历史数据。
- 原始证据保留。
- 精确重复与近重复检测。
- 跨来源事件聚类。
- 实体识别和消歧。
- 事件分类体系。
- 情绪、相关性、新颖度和重要性。
- 事件生命周期。
- 摘要和代表性证据。
- 因果或传导逻辑。
- 市场影响验证。
- 可解释性和审计。
- 对 QBot 的集成成本。

## 3. GDELT

参考：

- [GDELT Data and Documentation](https://www.gdeltproject.org/data.html)
- [GDELT 2.0: Our Global World in Realtime](https://blog.gdeltproject.org/gdelt-2-0-our-global-world-in-realtime/)
- [GDELT Terms of Use](https://www.gdeltproject.org/about.html#termsofuse)

### 3.1 架构特征

GDELT 提供两个主要结构：

- Event Database：使用 CAMEO 层级事件分类描述参与者、动作、地点和时间。
- Global Knowledge Graph：连接人物、组织、地点、主题、情绪、数量和来源。

GDELT 2.0：

- 约每 15 分钟更新。
- 处理多语言全球新闻。
- 将事件表和提及表分开。
- 同时提供原始批文件、BigQuery 和实时查询接口。

### 3.2 可借鉴点

- **事件与提及分离**：同一个现实事件可以有多个报道提及，每条提及保留来源和时间。
- **开放分层数据**：原始记录、标准事件和知识图谱各自独立。
- **明确分类体系**：事件类型是稳定代码，不是每次由模型自由生成。
- **新闻量归一化**：长期分析必须考虑不同时期媒体覆盖量变化。
- **全球宏观补充**：适合战争、制裁、选举、灾害、能源和供应链等跨境事件。

### 3.3 限制

- CAMEO 主要面向政治和社会行为，不是公司财务或产业链事件本体。
- 全球覆盖带来大量噪音和重复。
- 元数据不能替代可靠的公司公告或监管原文。
- GKG 可用于发现实体和主题，但不应直接视为经过验证的金融因果关系。

### 3.4 对 QBot 的定位

GDELT 应作为：

- 全球宏观和地缘事件补充源。
- 事件热度和跨国传播观察源。
- 研究和回放数据源。

不应作为：

- A 股公司公告的主数据源。
- 直接交易信号。
- 唯一事实核验来源。

## 4. Event Registry / NewsAPI.ai

参考：

- [Event Registry](https://eventregistry.org/)
- [NewsAPI.ai](https://www.newsapi.ai/)

### 4.1 架构特征

Event Registry 将新闻处理为两个层次：

1. Article：标题、正文、URL、发布时间、来源、实体、主题、情绪和社交传播。
2. Event：按内容相似度将描述同一现实事件的文章聚合。

平台还区分：

- 精确或近重复文章。
- 语义上属于同一事件、但内容不同的文章。
- 事件在不同来源和语言中的覆盖。

### 4.2 可借鉴点

- **重复检测与事件聚类必须分开**。复制稿不等于独立证据；不同角度的原创报道可以属于同一个事件。
- **实体消歧先于事件影响分析**。相同名称必须映射到规范公司、地点或人物。
- **保留文章层和事件层**。不能在聚类后丢失原文、来源和发布时间。
- **事件可跨语言和来源聚合**。
- **来源重要性、社交传播和报道数量是事件排序特征，但不是事实可信度本身**。

### 4.3 限制

- 属于通用新闻智能，不提供完整的 A 股产业传导逻辑。
- 商业接口需要评估许可、历史回溯和正文存储权限。
- 事件聚类能回答“哪些文章说的是同一件事”，不能自动证明事件如何影响利润和估值。

### 4.4 对 QBot 的定位

可以作为可选商业采集适配器。即使不采购，也应复制其数据模型：

```text
ArticleEvidence
NearDuplicateGroup
EventCluster
EntityMention
```

## 5. RavenPack News Analytics

参考：

- [RavenPack News Analytics](https://www.ravenpack.com/products/edge/data/news-analytics)

### 5.1 架构特征

RavenPack 的公开产品说明强调：

- 大规模实体库和事件分类体系。
- 情绪分析。
- 相关性评分。
- 新颖度跟踪。
- 时间评分。
- 事件影响分析。
- 公司、市场、行业和宏观用途。

### 5.2 最重要的设计模式

RavenPack 的核心不是“给一篇文章一个正负情绪”，而是对文章中识别到的每个实体和事件分别提供结构化分析。

QBot 应采用类似的关系模型：

```text
EventEntityLink
- event_id
- entity_id
- role
- relevance
- direction
- novelty
- effective_time
- expected_horizon
- confidence
```

同一篇新闻可能：

- 对上游公司为正面。
- 对下游公司为负面。
- 对文中顺带提及的公司相关性很低。

因此情绪和影响必须绑定到实体和角色，而不是绑定整篇文章。

### 5.3 可借鉴点

- 相关性、情绪、新颖度和时间维度独立建模。
- 新颖度需要和历史事件、重复报道及已有市场预期比较。
- 事件分类必须版本化。
- 分析结果可以作为量化因子，但必须与市场数据结合验证。

### 5.4 限制

- 商业闭源，无法复制其模型或分类细节。
- 外部评分仍然不能替代 QBot 自己的证据、逻辑链和回测。
- 商业声明不能直接作为策略有效性的证明。

## 6. Bloomberg NSTM / Key News Themes

参考：

- [NSTM: Real-Time Query-Driven News Overview Composition at Bloomberg](https://arxiv.org/abs/2006.01117)

### 6.1 生产架构

NSTM 使用：

1. 新闻采集时进行低成本在线增量聚类。
2. 查询时检索匹配文章及其在线聚类 ID。
3. 对已检索聚类再做层次聚类，修正在线聚类碎片化。
4. 为每个最终主题生成极短摘要和代表性文章。
5. 对主题排序并缓存结果。

### 6.2 可借鉴点

#### 双层聚类

第一层追求吞吐和低延迟，允许产生碎片；第二层在收盘批处理或用户查询时追求质量。

QBot 可以采用：

```text
采集时：增量候选聚类
收盘时：时间、实体、语义和动作联合精细聚类
```

#### 重要性不只看文章数量

NSTM 同时考虑：

- 聚类规模。
- 来源熵，即报道来源是否足够多样。

这可以减少同一媒体集团大量转载造成的虚假热度。

#### 代表性证据

每个事件应保存：

- 主要原始来源。
- 最具代表性的报道。
- 不同立场或新增信息的报道。

#### 可控摘要

NSTM 选择抽取式和受控压缩，而不是完全自由生成。QBot 的事实摘要也应优先从有证据的主谓宾和关键数字生成，再允许 LLM 润色。

### 6.3 对 QBot 的简化

QBot 第一版不需要亚秒查询和大型缓存，但应保留双层聚类接口，使未来扩展不需要改数据模型。

## 7. Dataminr 类实时事件检测

参考：

- [Real-time Event Detection on Social Data Streams](https://arxiv.org/abs/1907.11229)

该研究把实时事件表示为随时间变化的趋势实体聚类，并处理持续流入的大规模数据。

### 7.1 可借鉴点

- 事件不是固定文本，而是持续变化的聚类。
- 事件需要记录首次发现、最新更新和活跃度。
- 新证据可能改变事件严重度、地点、参与方和方向。
- 告警应区分“首次出现”和“已有事件的重要更新”。

建议生命周期：

```text
emerging -> active -> confirmed -> cooling -> closed
                    \-> contradicted
```

### 7.2 限制

QBot 不是大型实时公共安全平台，第一版没有必要复制其流计算规模。采用数据库状态机和周期任务即可。

## 8. 中文金融公告事件抽取

参考：

- [Doc2EDAG: An End-to-End Document-level Framework for Chinese Financial Event Extraction](https://arxiv.org/abs/1904.07535)
- [Knowledge Graph Enhanced Event Extraction in Financial Documents](https://arxiv.org/abs/2109.02592)

### 8.1 关键发现

公司公告中的一个事件经常具有：

- 多个参数分散在不同句子。
- 同一文档包含多个事件。
- 关键字段没有明确触发词。
- 公司、交易对手、金额、日期和标的之间存在关系约束。

因此公告处理不能仅依赖：

- 标题关键词。
- 单句分类。
- 通用情绪分析。

### 8.2 对 QBot 的设计

事件抽取分两条轨道：

#### 正式文档轨道

面向交易所、监管和上市公司公告：

- 文档级事件抽取。
- 固定事件 Schema。
- 数字、日期、主体和客体校验。
- 强来源可信度。
- 可直接形成事实节点。

#### 新闻报道轨道

面向媒体和聚合源：

- 文章聚类。
- 来源交叉验证。
- 事实、引用和媒体判断分离。
- 未确认内容保持不确定状态。

两条轨道最终进入同一个事件组，但保留不同证据等级。

## 9. 因果知识图谱

参考：

- [Constructing and Interpreting Causal Knowledge Graphs from News](https://arxiv.org/abs/2305.09359)
- [Talking to GDELT Through Knowledge Graphs](https://arxiv.org/abs/2503.07584)

### 9.1 成熟研究模式

因果知识图谱通常分两步：

1. 从文本抽取因果关系。
2. 对相似原因和结果参数聚类、规范化并连接成图。

研究结果支持结合：

- 模式和规则抽取。
- BERT 或 LLM 抽取。
- 参数聚类。
- 预定义本体。

### 9.2 为什么不能纯 LLM 生成逻辑链

近期对 GDELT 的研究发现，LLM 自动生成的知识图谱能够抓住摘要，但经常缺少一致性和可解释性；基于本体的图更稳定，但自动检索相关子图也有难度。

QBot 因此应采用：

```text
LLM 提取候选事实和关系
-> JSON Schema 校验
-> 实体规范化
-> 规则和产业链本体验证
-> 生成逻辑图
-> LLM 只负责将已存在的图转写为文字
```

每条边必须保存：

- 关系类型。
- 来源证据。
- 生成方式。
- 置信度。
- 允许成立的前提。
- 反证和失效条件。

## 10. 市场影响验证

成熟新闻产品通常提供影响分，但 QBot 需要可回测的自有验证口径。

### 10.1 事件研究

事件影响不能使用简单原始收益：

```text
股票涨 3% != 事件贡献 3%
```

至少需要计算：

- 股票相对市场异常收益。
- 股票相对行业异常收益。
- 行业相对市场收益。
- 成交额异常。
- 板块上涨宽度。
- 事件前是否已有预期交易。

### 10.2 事件窗口

按事件类型区分：

- 即时事件：`[0, 1]`、`[0, 3]` 个交易日。
- 政策和产业事件：`[0, 5]`、`[0, 20]`。
- 基本面长期事件：需要额外跟踪盈利和经营指标，不能只用长期股价归因。

### 10.3 混杂事件

同一窗口内出现财报、停复牌、监管或宏观冲击时，事件影响应标记为混杂，不能给出高置信因果结论。

### 10.4 历史事件类型基线

对每类事件建立历史分布：

```text
sample_count
median_abnormal_return
positive_rate
turnover_response
sector_breadth_response
time_to_peak
failure_rate
```

当前事件的“影响分”应表示其相对同类历史事件的位置，而不是模型主观感觉。

## 11. 流处理基础设施

参考：

- [Apache Flink CEP](https://nightlies.apache.org/flink/flink-docs-stable/docs/libs/cep/)

Flink CEP 可以在无限事件流上匹配有时间约束的复杂序列，并处理事件时间和迟到数据。它适合高吞吐、低延迟和大量有状态规则。

QBot 当前不需要引入 Flink，原因：

- 第一版主要按 30-60 分钟采集和收盘分析。
- 单实例 PostgreSQL 足以保存任务状态和事件生命周期。
- 引入 Kafka/Flink 会增加部署、监控、Schema 演进和故障恢复成本。

应保留升级条件：

- 每日证据量达到百万级。
- 分钟内低延迟告警成为核心要求。
- 单机任务无法按时完成。
- 多消费者需要独立重放事件流。

达到条件后，再将 `EventEvidence` 发布到消息流，并保持数据库合同不变。

## 13. 推荐的 QBot 事件架构

### 13.1 数据层次

```text
SourceItem
-> EventEvidence
-> DuplicateGroup
-> EventMention
-> EventCluster
-> EventEntityLink
-> EventLogicGraph
-> EventMarketObservation
-> EventTypeStatistics
-> DailyMarketBrief
```

### 13.2 关键区别

#### `DuplicateGroup`

复制或近复制内容。多条复制稿只算一个独立证据源。

#### `EventCluster`

描述同一现实事件的不同内容。可以包含多个独立来源和不同角度。

#### `EventEntityLink`

事件对某个公司、行业、商品或宏观变量的具体关系。方向和相关性在此层计算。

#### `EventLogicGraph`

只保存经过 Schema、实体映射和规则验证的逻辑节点和边。

#### `EventMarketObservation`

按交易日保存市场是否确认、否定或尚未反应。

### 13.3 两阶段聚类

```text
采集阶段：
URL / 哈希 / 近重复 + 轻量语义候选聚类

收盘阶段：
实体 + 动作 + 时间 + 地点 + 语义精细聚类
```

精细聚类允许：

- 合并碎片。
- 拆分错误大簇。
- 将更新稿挂到已有事件。
- 保留人工锁定关系。

### 13.4 双轨抽取

```text
正式公告：文档级固定 Schema 抽取
一般新闻：事实/引用/判断抽取 + 跨源确认
```

### 13.5 评分层次

禁止一个总分覆盖所有含义。

分别保存：

- `source_confidence`
- `evidence_independence`
- `entity_relevance`
- `event_novelty`
- `event_importance`
- `logic_confidence`
- `market_confirmation`
- `crowding_score`

最终日报和选股模块根据用途组合这些字段。

## 14. 推荐来源策略

### 第一层：直接事实源

- 交易所和上市公司公告。
- 监管机构和政府部门。
- 央行、统计、海关和行业主管部门。
- 商品和金融交易所。

这些来源用于创建高可信事实节点。

### 第二层：新闻聚合源

- GDELT：全球宏观、地缘和舆情补充。
- Event Registry / NewsAPI.ai：若许可和预算合适，用于全量新闻、正文和事件聚类。
- RavenPack：若未来需要成熟金融事件因子并有预算，可作为外部对照或补充，不能替代自有逻辑链。

### 第三层：人工输入

- Telegram。
- REST API。
- 人工修订、合并、拆分和优先级。

### 第一版建议

- 先实现官方公告/政策源 + GDELT + 人工输入。
- 保留商业 `NewsSource` 适配器接口。
- 不在核心逻辑中绑定某个商业供应商。
- 只保存许可允许持久化的正文；其他来源保存元数据、摘要、哈希和链接。

## 15. 修订后的实施顺序

### Phase 0：合同与测试数据

- 定义事件、证据、提及、实体关系、逻辑边和市场观察 Schema。
- 建立 50-100 个真实历史事件的黄金测试集。
- 明确内容许可和保留规则。

### Phase 1：证据仓库

- 人工输入。
- 一个官方公告或政策源。
- GDELT 宏观源。
- 精确和近重复检测。
- 来源等级和审计。

### Phase 2：事件聚类

- 增量候选聚类。
- 收盘精细聚类。
- 代表性证据和来源熵。
- 事件生命周期。

### Phase 3：结构化抽取

- 新闻事实 Schema。
- 公告文档级 Schema。
- 实体消歧。
- `EventEntityLink` 评分。

### Phase 4：逻辑链

- 五类确定性模板。
- 产业链和公司角色映射。
- 边级证据、前提和失效条件。
- LLM 文案渲染。

### Phase 5：市场验证

- 市场和行业异常收益。
- 成交和宽度。
- 事件类型历史基线。
- 确认、矛盾和过期状态。

### Phase 6：与选股连接

- 只将已确认的 `EventEntityLink` 暴露为有限权重因子。
- 事件不能单独把股票提升到 A 档。
- 记录事件因子对最终得分的边际贡献。

## 16. 最终架构决定

QBot 不复制某一家产品，而是组合成熟框架最可靠的部分：

- GDELT 的事件与提及分层。
- Event Registry 的去重、实体消歧和事件聚类。
- RavenPack 的实体级相关性、新颖度和影响维度。
- Bloomberg 的双层聚类、来源熵和代表性证据。
- Dataminr 的事件生命周期。
- Doc2EDAG 的公告文档级抽取。
- 因果知识图谱的混合抽取与确定性本体。
- 金融事件研究的异常收益验证。

最终原则：

> 新闻不是事实表，情绪不是影响，相关性不是因果，市场上涨也不自动证明逻辑正确。每一层都必须保留独立数据、证据和验证状态。
